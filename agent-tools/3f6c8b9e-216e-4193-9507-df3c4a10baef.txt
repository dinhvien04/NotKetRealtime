PHẦN 11  PERFORMANCE / OPTIMIZATION
================================================================================

11.1. Database
- Use `pg` Pool.
- Runtime `DATABASE_URL` should be pooled Neon URL.
- Migration `MIGRATION_DATABASE_URL` should be direct URL if available.
- Index hot queries.
- Use cursor pagination for messages.
- Avoid loading all messages.
- Avoid N+1 queries in conversation list.
- Use `LEFT JOIN LATERAL` carefully and index.
- Add query timeout if possible.
- Keep pool max reasonable, e.g. 10.

11.2. Socket.IO
- Debounce typing events client side.
- Rate limit message events.
- Room per user: `user:${id}`.
- Room per conversation optional: `conversation:${id}`.
- For scale:
  - Add Redis adapter optional.
  - Presence should move from memory to Redis.
  - Typing/presence TTL in Redis.
- Add health route `/health`.

11.3. Frontend
- Do not re-render entire message list on each new message.
- Deduplicate message by id.
- Virtualize messages later if very long.
- Lazy load images.
- Use thumbnails.
- Object URL preview should be revoked after use.
- Keep CSS static.
- Avoid big client bundle.

11.4. Upload
- Keep max upload 6MB default.
- For larger uploads later, use resumable upload/TUS.
- Compress image optional.
- Generate thumbnails.
- Use signed URLs cached until expiry.
- Do not proxy downloads through Node unless access control requires it.

11.5. AI
- Rate limit per user.
- Limit prompt length.
- Store AI sessions.
- Do not send full chat history blindly; cap context.
- Add timeout for AI provider calls.
- Retry only safe transient errors.

PHẦN 1  FIX NGAY CÁC LỖI BẢO MẬT / DATA-INTEGRITY HIỆN TẠI
================================================================================

1.1. Sửa authorization của `private_message`
Hiện socket event `private_message` có thể nhận cả `conversationId` và `receiverId`. Nếu client cố tình gửi `conversationId` của cuộc A-B nhưng `receiverId` là C, server không được emit message tới C.

Yêu cầu sửa:
- Nếu payload có `conversationId`:
  - Check user hiện tại là participant của conversation.
  - Lấy receiver thật bằng `conversationRepository.getOtherParticipant(conversationId, currentUser.id)`.
  - Không tin `receiverId` từ client.
  - Nếu client có gửi `receiverId` thì verify nó trùng với receiver thật.
  - Nếu không trùng, reject `{ ok:false, error:"Người nhận không thuộc hội thoại này." }`.
  - Emit message tới `user:${otherParticipant.id}`.
- Nếu payload không có `conversationId`:
  - Require `receiverId`.
  - Verify receiver tồn tại.
  - Tạo hoặc reuse direct conversation bằng `findOrCreateDirectConversation(senderId, receiverId)`.
- Không bao giờ lấy senderId từ client. Sender luôn là `socket.data.user.id`.

1.2. Sửa `mark_read`
Hiện `mark_read` chỉ check user là participant rồi update `last_read_message_id`.
Yêu cầu:
- Check `conversationId` tồn tại.
- Check user hiện tại là participant.
- Check `messageId` tồn tại.
- Check message đó thuộc đúng `conversationId`.
- Không cho mark read message của conversation khác.
- Nếu message không thuộc conversation, reject.
- Thêm test cho case cố tình mark read message của conversation khác.

1.3. Chống race condition khi tạo direct conversation
Hiện có bảng `direct_conversations` với unique `(user_low, user_high)`, nhưng application logic có thể race khi 2 request tạo cùng lúc.

Yêu cầu:
- Refactor `findOrCreateDirectConversation(userA, userB)`:
  - Chuẩn hóa `userLow`, `userHigh`.
  - Dùng transaction.
  - Dùng `INSERT ... ON CONFLICT (user_low, user_high) DO NOTHING RETURNING conversation_id`
  - Nếu conflict xảy ra, SELECT lại conversation_id.
  - Không để concurrent request crash.
- Thêm test hoặc service-level test cho duplicate/reuse direct conversation.

1.4. Sửa upload không phụ thuộc presence
Hiện `/api/uploads` đang bắt buộc `presenceModel.isOnline(sender.id)`.
Yêu cầu:
- Upload chỉ cần user đã login qua `requireAuth`.
- Không phụ thuộc socket online.
- Pending upload vẫn gắn theo `userId`.
- Khi socket gửi file message, mới consume pending upload.
- Nếu pending upload hết hạn hoặc không thuộc user, reject.

1.5. Thêm magic-byte validation cho upload
Hiện code chỉ tin `req.file.mimetype`. MIME từ client có thể fake.

Yêu cầu:
- Cài `file-type`.
- Với file upload:
  - Kiểm MIME từ Multer để reject sớm.
  - Kiểm magic bytes bằng `file-type` với buffer.
  - Với text/plain, có thể fallback kiểm buffer text an toàn.
  - Với doc/docx/xls/xlsx/ppt/pptx/pdf/image, verify file signature hợp lý nếu package hỗ trợ.
  - Nếu magic type không khớp allowlist thì reject.
- Không cho upload `.html`, `.svg`, `.js`, `.exe`, `.sh`, `.bat`, `.cmd`, `.php`, `.zip` trong bản

3.1. Auth hiện có
Giữ:
- register
- login
- logout
- me
- JWT HttpOnly cookie
- Argon2id

Cải thiện:
- rate limit login/register theo IP + username/email.
- Không tiết lộ username/email tồn tại ở forgot password.
- Chuẩn hóa error message login.
- Cập nhật `last_seen_at` và `updated_at`.

3.2. Register
API:
- POST `/api/auth/register`

Validation:
- username 3-40 ký tự.
- Cho phép chữ, số, gạch dưới, khoảng trắng đơn nếu muốn.
- Không cho HTML/tag.
- email optional nhưng nếu có phải đúng format.
- password tối thiểu 8 ký tự, tối đa 128 ký tự.
- confirm password phải trùng.
- Check duplicate username/email case-insensitive.

3.3. Login
API:
- POST `/api/auth/login`
- body: usernameOrEmail, password.
- Nếu account `is_locked` hoặc `status != active`, reject.
- Khi login thành công:
  - set cookie auth.
  - rotate session token nếu sau này có session table.
  - audit log action `auth.login`.

3.4. Logout
API:
- POST `/api/auth/logout`
- Require CSRF.
- Clear cookie.
- Audit log `auth.logout`.

3.5. Profile
APIs:
- GET `/api/users/me`
- PATCH `/api/users/me`
- POST `/api/users/me/avatar`
- POST `/api/users/me/change-password`

Profile fields:
- displayName
- bio
- avatarUrl
- email

Rules:
- User chỉ sửa profile của chính mình.
- Avatar upload dùng Supabase Storage, kind image only.
- Change password require old password.
- Change password hash Argon2id params chuẩn.
- Sau change password update `password_changed_at`.
- Có thể invalidate old tokens bằng cách check `iat > password_changed_at`.

3.6. Forgot password OTP
APIs:
- POST `/api/auth/forgot-password`
- POST `/api/auth/verify-reset-otp`
- POST `/api/auth/reset-password`

Flow:
1. User nhập email.
2. Server luôn trả message chung: Nếu email tồn tại, mã OTP đã được gửi.
3. Nếu email tồn tại:
   - Generate OTP 6 chữ số.
   - Hash OTP bằng HMAC/Argon2 hoặc SHA256 + secret pepper.
   - Store hash, expires 10 phút.
   - Rate limit request theo IP/email.

1.1. Sửa authorization của `private_message`
Hiện socket event `private_message` có thể nhận cả `conversationId` và `receiverId`. Nếu client cố tình gửi `conversationId` của cuộc A-B nhưng `receiverId` là C, server không được emit message tới C.

Yêu cầu sửa:
- Nếu payload có `conversationId`:
  - Check user hiện tại là participant của conversation.
  - Lấy receiver thật bằng `conversationRepository.getOtherParticipant(conversationId, currentUser.id)`.
  - Không tin `receiverId` từ client.
  - Nếu client có gửi `receiverId` thì verify nó trùng với receiver thật.
  - Nếu không trùng, reject `{ ok:false, error:"Người nhận không thuộc hội thoại này." }`.
  - Emit message tới `user:${otherParticipant.id}`.
- Nếu payload không có `conversationId`:
  - Require `receiverId`.
  - Verify receiver tồn tại.
  - Tạo hoặc reuse direct conversation bằng `findOrCreateDirectConversation(senderId, receiverId)`.
- Không bao giờ lấy senderId từ client. Sender luôn là `socket.data.user.id`.

1.2. Sửa `mark_read`
Hiện `mark_read` chỉ check user là participant rồi update `last_read_message_id`.
Yêu cầu:
- Check `conversationId` tồn tại.
- Check user hiện tại là participant.
- Check `messageId` tồn tại.
- Check message đó thuộc đúng `conversationId`.
- Không cho mark read message của conversation khác.
- Nếu message không thuộc conversation, reject.
- Thêm test cho case cố tình mark read message của conversation khác.

1.3. Chống race condition khi tạo direct conversation
Hiện có bảng `direct_conversations` với unique `(user_low, user_high)`, nhưng application logic có thể race khi 2 request tạo cùng lúc.

Yêu cầu:
- Refactor `findOrCreateDirectConversation(userA, userB)`:
  - Chuẩn hóa `userLow`, `userHigh`.
  - Dùng transaction.
  - Dùng `INSERT ... ON CONFLICT (user_low, user_high) DO NOTHING RETURNING conversation_id`
  - Nếu conflict xảy ra, SELECT lại conversation_id.
  - Không để concurrent request crash.
- Thêm test hoặc service-level test cho duplicate/reuse direct conversation.

1.4. Sửa upload không phụ thuộc presence
Hiện `/api/uploads` đang bắt buộc `presenceModel.isOnline(sender.id)`.
Yêu cầu:
- Upload chỉ cần user đã login qua `requireAuth`.
- Không phụ thuộc socket online.
- Pending upload vẫn gắn theo `userId`.
- Khi socket gửi file message, mới consume pending upload.
- Nếu pending upload hết hạn hoặc không thuộc user, reject.

1.5. Thêm magic-byte validation cho upload
Hiện code chỉ tin `req.file.mimetype`. MIME từ client có thể fake.

Yêu cầu:
- Cài `file-type`.
- Với file upload:
  - Kiểm MIME từ Multer để reject sớm.
  - Kiểm magic bytes bằng `file-type` với buffer.
  - Với text/plain, có thể fallback kiểm buffer text an toàn.
  - Với doc/docx/xls/xlsx/ppt/pptx/pdf/image, verify file signature hợp lý nếu package hỗ trợ.
  - Nếu magic type không khớp allowlist thì reject.
- Không cho upload `.html`, `.svg`, `.js`, `.exe`, `.sh`, `.bat`, `.cmd`, `.php`,
